# Live Hotel Finder プロダクト計画書

> 作成: 2026-07-06 ／ 4役ディスカッション（マーケター・PdM・デザインディレクター・エンジニアディレクター）による
> 「創造→構成→破壊→整える」3周の議論から策定

## 1. サービスの再定義

### 当初の仮説
「推し活で遠征するためのホテル探しを直感的にする」= 会場周辺ホテルの**価格比較**サイト

### 議論後の再定義
主戦場を「価格比較」から「**会場起点の宿泊意思決定支援**」へ重心移動する。

理由:
- 価格比較は楽天トラベルしか実価格APIが取得できず（じゃらんAPI終了・agoda要審査）、構造的に勝てない
- 実ユーザーの声の核心は「**目的地からの実質的な距離（移動時間）をベースに最適な宿を知りたい**」
  = 価格ではなく意思決定のニーズ
- 「終演後23時に会場を出て、歩いて帰れて、翌朝チェックアウトが遅い宿」という文脈は
  既存OTA（じゃらん・トリバゴ等）に存在しない検索軸

### 独自価値仮説（要検証）
> 「安い宿が見つかる」ではなく「**終演後の自分が救われる宿が、考えなくても分かる**」

## 3. 実施フロー（フェーズとゲート）

```
Phase 0│準備
       │ □ 調査ノート(docs/research/)をリポジトリに作成 ✅
       │ □ インタビュー台本作成（過去の行動のみを聞く／意見を聞かない）✅
       │ □ Xリスニングの検索クエリ確定 ✅
       ▼
Phase 1│発見調査（1〜2週目）
       │ □ デスクリサーチ（市場規模・競合・ペインの生の声）✅ → docs/research/desk-research.md
       │ □ 競合(じゃらん/Googleマップ/トリバゴ)で自己予約体験 → 不満点記録
       │ □ 遠征経験者5人インタビュー + プレビュー操作観察 ← 【人間が実施】
       ▼
 GATE 1│「移動時間ベースの意思決定」ジョブは実在するか？
       │ YES → Phase 2 ／ NO → ジョブ再定義に戻る（開発しない）
       ▼
Phase 2│定量検証（3週目）
       │ □ アンケートn=50でペイン優先順位を定量化 ← 【人間が配布】
       │ □ 検証済みペイン → 機能要件に翻訳（docs/requirements-v3.md）
       ▼
 GATE 2│上位ペインと下記ロードマップは一致しているか？ ズレたら並べ替え
       ▼
Phase 3│開発（4週目〜）ブランチ: claude/hotel-price-api-integration
       │ ① 終演後シミュレーター（終演時刻→徒歩圏/終電圏の色分け）
       │ ② 楽天トラベルAPI実価格 + 実在ホテル自動取得
       │ ③ アフィリエイトURL化（楽天/A8/agoda、env未設定時フォールバック）
       ▼
Phase 4│公開検証（6週目〜）
       │ □ インタビュー協力者に本番URLを配布 → 次の遠征で実利用してもらう
       │ □ KPI: 「検索→OTAクリック率」と「再訪」のみ（収益はまだ見ない）
```

## 4. 機能ロードマップ（優先順位確定版）

| 順位 | 機能 | 対応ジョブ | 実装コスト | 備考 |
|---|---|---|---|---|
| 1 | 終演後シミュレーター（徒歩圏色分け） | J1直撃 | 低（既存データで可能） | MVP完成の核 |
| 2 | 楽天API実価格 | 信頼性 | 中 | アフィリエイトIDとセット |
| 3 | アフィリエイトURL化 | 収益 | 低 | env未設定時は素URLへフォールバック |
| 4 | 公演起点検索 | 集客 | 中 | 公演データ源が必要・**MVPから除外** |
| 5 | 会場数拡大 20→50 | カバレッジ | 低〜中 | 楽天API接続**後**（実在ホテル自動取得とセット） |
| 6 | しおり共有・高騰アラート | J5/リテンション | 高 | 次期 |

## 5. 決定事項（破壊フェーズで確定した制約）

1. **収益はKPIにしない**: ホテルアフィリエイト料率は約1%。当面は「使われるか」
   （検索→OTAクリック率・再訪）だけを見る。
2. **公演データはMVPに入れない**: ライセンスと更新運用が重く、MVPを殺すため。
3. **会場拡大はAPI接続後**: ダミーホテルのまま会場を増やしても意味が薄い。
4. **「1行=1ホテル」の情報密度は聖域**: 機能追加でリスト密度を崩さない。
5. **インタビューはMom Test方式**: 「このアプリ良いと思う?」と絶対に聞かない。
   過去の実際の遠征行動だけを聞く。
